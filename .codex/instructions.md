
# Overview

This project is a C# implementation of a graphic tool for analysis of flow cytometry
data, in the form of FCS files. FCS files contain annotated columns of channels, each
containing values of data points. Representing high dimension (ranging from 1 to around 200)
data. The project is base upon C# MVVM architecture on Avalonia.

This project should include:
- Workspace system and groupings. Samples that have exactly the same channel profiles
  can be placed under one group. Each group have shared gating schemes, statistics definitions,
  compensation matrices, and can append new samples after being created and configured. 
  Newly added samples should automatically adapt to the group's configuration. Samples do not
  contain individual schemes themselves, but according to the groupings.
- Visualizer system for populations in 2D. And editable gating controls. This will be explained 
  further in the following sections.

## Project system

- A project consists of several groupings.
- A grouping consists of several samples with *matched* channels. Groupings hosts schemes.
  A flow cytometry sample group hosts the following schemes:
  - Gating strategy tree. This defines the generation of subsets by gating strategies.
    - Gating strategies contains six types: polygonal gating, rectangular gating, 
      threshold gating, range gating, quadrant gating, and curly quadrant gating.
      Each gating is recalled by individual samples to generate real-time update populations.
    - Gating strategies are defined by coordinates (always transformed to global linear coordinates
      for points) and coordinate systems (Linear, Logicle with parameters). Points should be
      transformed since the representation is equal. However edges (How to connect those vertex) 
      are defined in respective coord systems. For example, a polygon defined in linear system
      have curly edges in the logicle system. and vice versa. This should be reflected on the
      plotting of the gates.
    - Gating commonly involves two axes to be defined (e.g. FSC-A and SSC-A). 
    - Several gating strategies depends on only one axis, In this case, plot using histogram.
  - Statistical definitions. Statistical definitions are defined in the gating strategy.
    This is recalled by samples to use that specific population defined by the gate to calculate
    statistics.
    - Supported statistics are:
      - Mean
      - Median
      - Geometric mean
      - Coefficient of variation (CV)
      - Standard deviation (SD)
      - Frequency of parent population
      - Frequency of all
      - Number of events
  - Lower dimensional embeddings: are defined in gating strategy, produced by dimensionality 
    reduction algorithms. Individual algorithms can leave unimplemented.
  - Compensation matrices: are defined in grouping. Records the compensation matrix for configuration
    of channels.
- A sample is loaded from FCS files, or generated by code. A sample contains channel definitions and
  an array of multi-dimensional data points. It should respond to configuration changes in the 
  groupings where there are changes to the gating, statistical definitions, and compensation matrices.
  It should update its subpopulation tree to accord with the grouping's gating strategy, and 
  recalculate statistics according to the strategy's definition. It should keep the raw counts,
  and compensated counts, and recalculate compensated counts when compensation matrix changes.
  Similarly, it should contains:
  - Subpopulations (according to gating schemes)
  - Statistics (according to statistical definitions)
  - Lower dimensional embeddings

The project system should be maintained as a tree data grid.
Avalonia.Controls.TreeDataGrid is installed as dependency.

Drags and drops of samples should be implemented. Selection and later deletions etc. operations
should be implemented.

## Plotting system

This is the core of the GUI. You should implement from scratch the plotter and gating editor for
flow cytometry data. It should be a X-Y plot when user specify a subpopulation or gating schemes.

The plotting modes
- (Default): 300x300 density map. Calculate the observed density onto the grid defined by the view
  (a mesh grid defined by max value, min value, and coordinate system). This ensures constant
  time usage when handling >1M cells. Never use real scatter plots, the time consumption is 
  inacceptable. Should be plotted using colormap of the density.
- Contour plot.
- Zebra plot.
- Histogram. Disable the y-channel, and plot X histograms. This is default when visualizing 
  one dimensional gates e.g. threshold or range.

Annotation modes
- Show grid
- Show gating annotation
- Show gating number of events
- Show gating population percentage of parent.

Visualizer
- The plotting system should visualize (1) Subpopulation of samples (2) Gating strategy
  When visualizing gating strategy, it should plot all available dots of the whole group
  under the gating scheme.
- Axis: should be configurable with min, max and coordinate system. currently supported:
  - Linear
  - Logicle with four parameters
  Min values should be default to 0, and max values be the maximum channel value.
- A toolbar of toogle buttons of gating tools. This activate the tool for creating and editing
  gating schemes. Existing gates should be editable. This is determined by the mouse position.
  The anchor points (vertex etc) should be a highlight bordered white square, move mouse hover,
  it should be stressed, and if mouse down, being solid highlighted. The points are transformable
  and editable in both coordinate system, but the edges should take respect of coordinate systems.
  It should be correctly rendered as curly lines if not in the defined coordinate.
  For threshold and range gates, the anchor points are anchor 'lines', the whole line is selectable.
  For quadrant cross and curly quadrant, the crossover point is the only anchor.
- Previous and next sibling nodes exchange buttons
- Axis should be (1) All channels in the group (2) All embedding dimensions in the population 
  strategy.

The visualizer should be build directly with canvas. Never try to import external packages 
for plotting. I need full control of it.

## Coding Styles

- All private or internal functions, fields, variables should be named as **snake-cased**.
  e.g. `private_var`. Replace to snake case everywhere you think to use small camel case.
  I do not allow small camel case in my code. 
- Do not add brace to control statements if the body is only one line.
